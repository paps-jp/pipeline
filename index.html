アイドルGPUを、
バッチで埋め尽くす

推論 / 学習 / 解析 — 複数のバッチワークロードを同じGPUフリートに同居させ、 self-loopでキューを絶やさず、残った空き時間を別workloadで埋める。 GPUを「ジョブ単独で半分しか使えない」から「常時ほぼフル稼働」へ。

セットアップを見る マニュアル

なぜGPUの空きを埋め切れるのか

単一ジョブで動かすGPUは、入力待ち / モデルロード / preprocess / 後処理で 案外半分くらいアイドルしている。Pipelineは4つの構造でその隙間を別workloadに 流し込んで、アイドル時間を残さない。

self-loopでキューを絶やさない

ワークロード自身が「次のtick」を自分でキューに入れ続ける設計。外部のdispatcher / cronが不要、キューが空になる瞬間が無いので、ワーカーは常に次の仕事を受け取れる。

1タスク = 1 claim、順序自由

ワーカーはキューから1件取って処理 → 完了 → 次の1件、を繰り返すだけ。バッチbarrier (= N件揃ってから次へ) が無いので、速いtaskと遅いtaskが同居しても速い方が先に流れてGPUを空かせない。

複数workloadの同居 + 重み配分

同じワーカーfleetにN個workloadを登録できる。優先度 + 重みをOptimizerが自動調整、「メインの推論70%、隙間の解析30%」のような同居構成を1 fleetで実現する。重い学習が来たら軽い推論を一時退避、終わり次第復帰。

GPUタグで必要なワーカーへ

ワーカーごとに gpu / cpu / 任意タグを付与。GPU必須のtaskはGPUワーカーへ、CPUでOKなtaskはCPUワーカーへアフィニティ振り分け。GPUは「GPUが本当に要るタスク」だけで埋まる。

主な特徴

GUIでワークロード登録

シェルコマンドやPythonモジュールをWeb UIのフォームから登録。開発者以外の運用担当でも追加・編集できる。

プラグイン方式

ローカル plugins/ ディレクトリに plugin.yaml + main.py を置けば即フォーム化。サンプル付属。

シンプルなワーカー

各ホストに1つのデーモンがHTTP heartbeatで参加。クラスタマネージャ不要、SSH不要 (= 配信はHTTP経由)。

ライブ可視化

ダッシュボードで実行中タスク / 最近の失敗 / キュー深さ / GPU温度を3秒更新で表示。

失敗詳細 + 再実行

各runのstderr / Python tracebackをワンクリックで展開。失敗したtaskは同じpkで即再投入できる。1件詰まって全体停止、というAirflow系の落とし穴を構造的に回避。

SQLiteシングルファイル

制御プレーンの全状態が1つのSQLiteファイルに収まる。バックアップ・差分・grepが容易。

アーキテクチャ

制御プレーン (= FastAPI + SQLite + React配信) とワーカー (= ジョブ実行プロセス) は 明確に分離されている。ワーカーはHTTP heartbeatで制御プレーンに参加し、キューから タスクをclaim → process → completeする。

画面ツアー

主要4画面のスクリーンショット。clickで原寸表示。

これでできること

シナリオ仕組み
定期的に1行 = 1タスクのSQLクエリを並列実行したい ワークロード1つ + ワーカーN台 → 自動分散
GPUを使うモデル推論を複数ホストで回したい Pythonプラグイン + ワーカーtags=gpuでアフィニティ
失敗したタスクだけを後で再実行したい RunsDrawerの各行に「再投入」ボタン
運用担当 (= 非エンジニア) がジョブ設定を変更したい plugin.yaml の init_kwargs からフォームを自動生成
新しいGPU箱を5分で追加したい bootstrapスクリプト1行 + Deployタブで配信先登録

これは向いていない

  • 10,000ノード以上の分散学習 — 専用フレームワーク (Rayなど) の方が適切
  • ノード間で大tensorをzero-copyで渡すワークロード — object storeが無いため
  • 複雑なDAG / Workflowオーケストレーション — 直列chain + self-loopのシンプル設計
  • マルチテナント (複数チーム共有) — 単一operatorを想定
