GUI から登録した バッチを
フリート全体で並列実行

Pipeline は ワークロードを Web UI から登録し、 複数ホストの ワーカープールで 実行する シンプルな バッチ制御プレーン。 SQLite + FastAPI + React 製。

セットアップを見る マニュアル

主な特徴

📋

GUI でワークロード登録

シェルコマンドや Python モジュールを Web UI のフォームから登録。 開発者以外の運用担当でも追加・編集できる。

🔌

プラグイン方式

ローカル plugins/ ディレクトリに plugin.yaml + main.py を置けば 即フォーム化。 サンプル付属。

⚡

シンプルな ワーカー

各ホストに 1 つのデーモンが HTTP heartbeat で参加。 クラスタマネージャ不要、 SSH 不要 (= 配信は HTTP 経由)。

📊

ライブ可視化

ダッシュボードで 実行中タスク / 最近の失敗 / キュー深さ / GPU 温度を 3 秒更新で表示。

🔄

失敗詳細 + 再実行

各 run の stderr / Python traceback を ワンクリックで展開。 失敗した task は 同じ pk で 即 再投入できる。

🎯

SQLite シングルファイル

制御プレーンの全状態が 1 つの SQLite ファイルに収まる。 バックアップ・差分・grep が容易。

アーキテクチャ

制御プレーン (= FastAPI + SQLite + React 配信) と ワーカー (= ジョブ実行プロセス) は 明確に分離されている。 ワーカーは HTTP heartbeat で制御プレーンに参加し、 キューから タスクを claim → process → complete する。

これでできること

シナリオ仕組み
定期的に 1 行 = 1 タスクの SQL クエリを 並列実行したい ワークロード 1 つ + ワーカー N 台 → 自動分散
GPU を使う モデル推論を 複数ホストで 回したい Python プラグイン + ワーカー tags=gpu でアフィニティ
失敗した タスクだけを 後で再実行したい RunsDrawer の各行に 「再投入」 ボタン
運用担当 (= 非エンジニア) が ジョブ設定を 変更したい plugin.yaml の init_kwargs から フォームを自動生成
新しい GPU 箱を 5 分で 追加したい bootstrap スクリプト 1 行 + Deploy タブで配信先登録

これは向いていない

  • 10,000 ノード以上の分散学習 — 専用フレームワーク (Ray など) の方が適切
  • ノード間で 大 tensor を zero-copy で渡す ワークロード — object store が無いため
  • 複雑な DAG / Workflow オーケストレーション — 直列 chain + self-loop の シンプル設計
  • マルチテナント (複数チーム共有) — 単一 operator を想定
