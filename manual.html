1. 用語と概念

用語説明
制御プレーン (Control plane) FastAPI + SQLiteで動く単一プロセス。Web UI配信 + REST API + ワーカーheartbeat受付。
ワーカー (Worker) 各ホストで動くデーモン。制御プレーンにheartbeatを10秒ごとに送り、キューからタスクを1つずつ取り出して処理。
ワークロード (Workload) 「何をどう実行するか」の定義。slugで一意。Web UIから登録する1つの単位。
キュー (Queue) ワークロード毎に1つ作られる、タスクの待ち行列。SQLite内の <slug>_queue テーブル。
タスク (Task) キューに1行入る処理単位。 pk (主キー) + extra (任意のJSON) を持つ。
Run タスクの実行履歴1件。成功 / 失敗 / 経過時間 / stdout / stderr / output_jsonが保存される。
Executor タスクを実際に処理する方式。 shell (= 任意コマンド) と python_module (= プラグイン呼出) の2種類。
プラグイン Pythonモジュール1つを plugins/<name>/ に配置。 plugin.yaml + setup()/process()/cleanup() をexportする。

1.1タスクのライフサイクル

  1. enqueue — Web UIまたはPOST APIでキューにpkを1件追加 (= state pending)
  2. claim — 空きのワーカーがclaim_batchで1件取得 (= state claimed、リース付き)
  3. process — ワーカーがプラグインの process(task) を呼ぶ
  4. complete / fail — 戻り値がdictなら成功 (= キュー行を削除 + runsに記録)、例外なら失敗 (= attempt増加 + state failed)
  5. retry — failedタスクはmax_attempts未満なら再度pendingに戻る。上限に達するとdead-letterとして残る。

2. Dashboard

制御プレーンの全体状態を一画面で把握するためのページ。3秒ごとに更新。

パネル内容
実行中finished_atがnullのrun一覧 (workload / pk / worker / 経過秒)
最近の失敗直近300件からsuccess=falseを抽出 (最大10件)、Python tracebackの最終行を表示
キュー深さenabled各workloadのpending / claimed / failedをバーで可視化
システム情報version / mode (= dev/prod) / DB URL / 現在時刻
GPU温度ワーカーがnvidia-smiメトリクスをheartbeatに乗せた場合、過去30分の温度をスパークラインで表示 (70℃ で黄、80℃ で赤)

3. Workloads

ワークロードの一覧 + 作成 + 編集。各行に「直近20件の成否」がスパークラインで表示される (緑 = 成功 / 赤 = 失敗)。

3.1ワークロード作成

「新規作成」ボタンでDrawerが右から開く。入力項目:

項目説明
slugURL/識別子。小文字英数 + ハイフン / アンダースコア。一度作ったら変更不可。
表示名UI上での人間可読名。
説明(任意) 何をするworkloadかメモ。
実行モードshell または python_module を選択。
実行設定shellならコマンド配列、python_moduleならプラグイン + モジュール + 設定。
優先度0–100。ワーカーが複数workloadに対応する場合高優先を先に処理。
重み同優先度内での配分比率 (= weighted fair queue)。
バッチサイズワーカーが1回のclaim_batchで取る件数。大きいほどスループット上昇、小さいほど公平性 / 失敗局所化。
lease秒数claim後この秒数workerからcomplete/failが来なければ他workerが再claim可能。
max_attemptsfail → retryの上限回数。超えるとdead-letterとして残る。

3.2タスク投入

各行の 📨 アイコンでモーダルを開き、pkと任意のextra (JSON) を入力して投入。同じpkは重複扱いされINSERT IGNOREされる。

3.3実行履歴

各行の履歴アイコンでRunsDrawerが右から開く。

  • キュー状態 (pending / claimed / failed) のバッジ表示
  • 直近50件のrunを表形式で表示
  • 各行クリックでstdout / stderr / error / output_jsonを展開 (= 失敗原因が一目で分かる)
  • 各行の 🔄 アイコンで 同じpkを再投入

3.4 URL deep link

各workloadに直接URLでアクセス可能:

  • /workloads/<slug> — そのworkloadの編集Drawerが開く
  • /workloads/<slug>/runs — RunsDrawerが開く

SlackやIssueにURLを貼る運用に便利。

4. Workers

接続中ワーカーの一覧。表示項目:

列意味
idワーカー識別子 (= 起動時に --worker-id で指定、デフォルトはhostname)
hostワーカーのホスト名
stateactive / draining / lost / connecting
current_workload現在処理中のworkload (claim中なら表示)

5. ServiceLog

制御プレーン + 各ワーカーホストのstdout/stderrをタブで切り替えて表示。動作中はtealのパルスバッジが点滅、一時停止中は灰色。

  • Pipelineタブ — 制御プレーン (= pipeline-oss-control.service) のログ
  • 各ホストタブ — Deployで登録した各配信先ホストのログ (= 配信時に該当ホストのservice stdout/stderrが制御プレーンに集約)

タブ切替時に自動で最下行 (= 最新) にスクロール。「一時停止」で自動更新を停止できる。

6. Deploy

制御プレーンから各ホストへファイルやコードを一括配信する仕組み。SSHを裏で使うが、オペレータはWeb UIだけで完結。

6.1配信先ホスト

SSH接続先の一覧:

  • label: 表示名
  • host: IPまたはFQDN
  • ssh_user / ssh_port: 通常 root / 22
  • enabled: チェックを外すとこの配信から除外される

「公開鍵を表示」で制御プレーンが使うSSH公開鍵を表示。これを配信先ホストの ~/.ssh/authorized_keys に入れることでパスワード無しの配信が可能になる。

6.2配信パス

「何をどこへ」の設定:

  • src_path: 制御プレーン上のパス (= 配信元)
  • dst_path: 配信先ホストでのパス (= 配信先)
  • setup_command: 配信直後に1回実行するコマンド (任意)
  • service_command: serviceとして常駐させる場合のコマンド (任意)
  • delete_mode: rsync --deleteを付ける (= 余分なファイルを配信先で削除)

6.3配信実行

「Deploy now」ボタン → 有効化された全配信先 × 全配信パスを一括実行。

  • skip restart: rsyncのみでservice restartは行わない
  • dry run: rsync --dry-runで何が配信されるかだけを表示

結果は履歴に残り、各runのログを展開して確認できる。

7.3 main.py

from typing import Any

def setup(**kwargs: Any) -> dict[str, Any]:
    """プロセス起動時に1回呼ばれる。重い初期化 (モデルロード / DB接続)
    はここで。戻りdictがprocess() にctxとして渡される。"""
    return {
        "api_url": kwargs["api_url"],
        "timeout": kwargs.get("timeout_secs", 10),
        "client": _make_http_client(),
    }

def process(task: Any, **ctx: Any) -> dict[str, Any]:
    """1タスク毎に呼ばれる。task.pk + task.extraを元に処理。
    戻りdictはruns.output_jsonに保存される。例外を上げると失敗扱い。"""
    response = ctx["client"].get(f"{ctx['api_url']}/{task.pk}", timeout=ctx["timeout"])
    return {"status_code": response.status_code, "size": len(response.content)}

def cleanup(**ctx: Any) -> None:
    """プロセス停止時に呼ばれる (任意)。接続のクローズなど。"""
    ctx["client"].close()

8. Executor種別

8.1 shell

任意のシェルコマンドを1タスク = 1プロセスとして実行する。

{
  "command": ["bash", "-c", "echo task={task.pk}"],
  "cwd": null,
  "env": {"MY_VAR": "value"},
  "timeout_secs": 60
}

テンプレート変数 {task.pk} {task.extra.KEY} が展開される。

8.2 python_module

プラグインを呼ぶ。instanceはworkerプロセスでキャッシュされるため、重いsetup() は1回だけ実行。

{
  "source_path": "plugins/my_plugin",
  "module": "main",
  "init_kwargs": {
    "api_url": "https://api.example.com",
    "timeout_secs": 10
  }
}

10. 設定リファレンス

10.1制御プレーンCLI

pipeline run [OPTIONS]

  --host TEXT        listenするホスト名/IP (デフォルト0.0.0.0)
  --port INTEGER     listenポート (デフォルト8001)
  --db-url TEXT      SQLAlchemy URL (例: sqlite:///./pipeline.db)
  --log-level TEXT   debug / info / warning / error
  --mode TEXT        dev / prod (= CORS設定が変わる)

10.2ワーカーCLI

pipeline worker [OPTIONS]

  --control-url TEXT制御プレーンのURL (例: http://10.0.0.1:8001) [必須]
  --worker-id TEXTワーカー識別子 (デフォルトhostname)
  --tags TEXTカンマ区切り (例: gpu,cuda12)
  --token TEXT制御プレーン認証トークン (任意)
  --log-level TEXTログレベル

10.3環境変数

変数意味
PIPELINE_DB_URL--db-url と同等
ADMIN_PASSWORD初回起動時に管理者パスワードを設定 (= 認証を有効化する場合)

11. 運用Tips

11.1バックアップ

SQLiteファイル (pipeline.db) をそのままコピーすればフルバックアップ。制御プレーンが動作中でもWALモードなので読み出しロックは発生しにくい。確実を期すなら VACUUM INTO:

sqlite3 pipeline.db "VACUUM INTO 'pipeline-backup-$(date +%Y%m%d).db'"

11.2ワーカー追加 / 削除

ワーカーはpull型なので、追加はデーモン起動だけでOK (= 制御プレーン側の設定変更不要)。削除はデーモン停止だけ、60秒後に lost、600秒後にDBから消える。

11.3ワーカーのアフィニティ

workloadの host_affinity を設定すると、タグが一致するワーカーだけがclaimできる。例: ["gpu"] を指定するとtags=gpu 持ちのみ。

11.4失敗のリトライ

例外で失敗したタスクは max_attempts 回まで自動retry。上限超過後はキューに残るがclaimされなくなる (= dead-letter)。これらはDashboardのキュー深さで failed として見える。手動で再投入するか、失敗原因を修正してenable切替で再開。

11.5監視

Prometheus形式のメトリクスendpointは未実装 (= 必要に応じて追加可能)。当面は:

  • Dashboardのキュー深さで滞留を監視
  • /api/v1/dashboard/overview の recent_failures を定期ping
  • nvidia-smiメトリクスはワーカーから自動収集
