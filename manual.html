1. 用語と概念

用語説明
制御プレーン (Control plane) FastAPI + SQLiteで動く単一プロセスです。Web UI配信 + REST API + ワーカーheartbeat受付を担います。
ワーカー (Worker) 各ホストで動くデーモンです。制御プレーンにheartbeatを10秒ごとに送り、キューからタスクを1つずつ取り出して処理します。
ワークロード (Workload) 「何をどう実行するか」の定義です。slugで一意となり、Web UIから登録する1つの単位です。
キュー (Queue) ワークロード毎に1つ作られる、タスクの待ち行列です。SQLite内の <slug>_queue テーブルとして保存されます。
タスク (Task) キューに1行入る処理単位です。 pk (主キー) + extra (任意のJSON) を持ちます。
Run タスクの実行履歴1件です。成功 / 失敗 / 経過時間 / stdout / stderr / output_jsonが保存されます。
Executor タスクを実際に処理する方式です。 shell (= 任意コマンド) と python_module (= プラグイン呼出) の2種類があります。
プラグイン Pythonモジュール1つを plugins/<name>/ に配置します。 plugin.yaml + setup()/process()/cleanup() をexportします。

1.1タスクのライフサイクル

  1. enqueue — Web UIまたはPOST APIでキューにpkを1件追加します (= state pending)。
  2. claim — 空きのワーカーがclaim_batchで1件取得します (= state claimed、リース付き)。
  3. process — ワーカーがプラグインの process(task) を呼びます。
  4. complete / fail — 戻り値がdictなら成功 (= キュー行を削除 + runsに記録)、例外なら失敗 (= attempt増加 + state failed) となります。
  5. retry — failedタスクはmax_attempts未満なら再度pendingに戻ります。上限に達するとdead-letterとして残ります。

2. Dashboard

制御プレーンの全体状態を一画面で把握するためのページです。3秒ごとに更新されます。

パネル内容
実行中finished_atがnullのrun一覧です (workload / pk / worker / 経過秒)。
最近の失敗直近300件からsuccess=falseを抽出 (最大10件) し、Python tracebackの最終行を表示します。
キュー深さenabled各workloadのpending / claimed / failedをバーで可視化します。
システム情報version / mode (= dev/prod) / DB URL / 現在時刻を表示します。
GPU温度ワーカーがnvidia-smiメトリクスをheartbeatに乗せた場合、過去30分の温度をスパークラインで表示します (70℃ で黄、80℃ で赤)。

3. Workloads

ワークロードの一覧 + 作成 + 編集を行うページです。各行に「直近20件の成否」がスパークラインで表示されます (緑 = 成功 / 赤 = 失敗)。

3.1ワークロード作成

「新規作成」ボタンでDrawerが右から開きます。入力項目は以下のとおりです。

項目説明
slugURL / 識別子です。小文字英数 + ハイフン / アンダースコアで指定します。一度作ったら変更できません。
表示名UI上での人間可読名です。
説明(任意) 何をするworkloadかのメモです。
実行モードshell または python_module を選択します。
実行設定shellならコマンド配列、python_moduleならプラグイン + モジュール + 設定を指定します。
優先度0–100で指定します。ワーカーが複数workloadに対応する場合、高優先を先に処理します。
重み同優先度内での配分比率です (= weighted fair queue)。
バッチサイズワーカーが1回のclaim_batchで取る件数です。大きいほどスループットが上がり、小さいほど公平性 / 失敗局所化が向上します。
lease秒数claim後この秒数workerからcomplete/failが来なければ、他workerが再claimできます。
max_attemptsfail → retryの上限回数です。超えるとdead-letterとして残ります。

3.2タスク投入

各行の 📨 アイコンでモーダルを開き、pkと任意のextra (JSON) を入力して投入します。同じpkは重複扱いとなり、INSERT IGNOREされます。

3.3実行履歴

各行の履歴アイコンでRunsDrawerが右から開きます。

  • キュー状態 (pending / claimed / failed) のバッジを表示します。
  • 直近50件のrunを表形式で表示します。
  • 各行をクリックするとstdout / stderr / error / output_jsonが展開され、失敗原因が一目で分かります。
  • 各行の 🔄 アイコンで 同じpkを再投入 できます。

3.4 URL deep link

各workloadに直接URLでアクセスできます。

  • /workloads/<slug> — そのworkloadの編集Drawerが開きます。
  • /workloads/<slug>/runs — RunsDrawerが開きます。

SlackやIssueにURLを貼る運用に便利です。

4. Workers

接続中ワーカーの一覧です。表示項目は以下のとおりです。

列意味
idワーカー識別子です (= 起動時に --worker-id で指定、デフォルトはhostname)。
hostワーカーのホスト名です。
stateactive / draining / lost / connecting のいずれかです。
current_workload現在処理中のworkloadです (claim中なら表示されます)。
processed累計処理タスク数です。
errors累計エラー数です。
last_seen最後のheartbeatからの経過時間です。
started_at起動時刻です。
stateの遷移: heartbeatが60秒来ないと lost となり、さらに600秒来ないと DBから削除されます (= worker_metricsの該当行もリーパーで削除されます)。

5. ServiceLog

制御プレーン + 各ワーカーホストのstdout/stderrをタブで切り替えて表示できます。動作中はtealのパルスバッジが点滅、一時停止中は灰色になります。

  • Pipelineタブ — 制御プレーン (= pipeline-oss-control.service) のログです。
  • 各ホストタブ — Deployで登録した各配信先ホストのログです (= 配信時に該当ホストのservice stdout/stderrが制御プレーンに集約されます)。

タブ切替時に自動で最下行 (= 最新) にスクロールします。「一時停止」で自動更新を停止できます。

6. Deploy

制御プレーンから各ホストへファイルやコードを一括配信する仕組みです。SSHを裏で使いますが、オペレータはWeb UIだけで完結できます。

6.1配信先ホスト

SSH接続先の一覧を登録します。項目は以下のとおりです。

  • label: 表示名です。
  • host: IPまたはFQDNです。
  • ssh_user / ssh_port: 通常は root / 22 です。
  • enabled: チェックを外すとこの配信から除外されます。

「公開鍵を表示」で制御プレーンが使うSSH公開鍵が表示されます。これを配信先ホストの ~/.ssh/authorized_keys に入れると、パスワード無しの配信ができるようになります。

6.2配信パス

「何をどこへ」の設定です。

  • src_path: 制御プレーン上のパスです (= 配信元)。
  • dst_path: 配信先ホストでのパスです (= 配信先)。
  • setup_command: 配信直後に1回実行するコマンドです (任意)。
  • service_command: serviceとして常駐させる場合のコマンドです (任意)。
  • delete_mode: rsync --deleteを付けます (= 余分なファイルを配信先で削除)。

6.3配信実行

「Deploy now」ボタンで、有効化された全配信先 × 全配信パスを一括実行できます。

  • skip restart: rsyncのみでservice restartは行いません。
  • dry run: rsync --dry-runで何が配信されるかだけを表示します。

結果は履歴に残り、各runのログを展開して確認できます。

7.2 plugin.yaml

name: my-plugin

init_kwargsのtype

typeWeb UIで表示されるinput

7.3 main.py

from typing import Any

def setup(**kwargs: Any) -> dict[str, Any]:
    """プロセス起動時に1回呼ばれます。重い初期化 (モデルロード / DB接続)
    はここで行います。戻りdictがprocess() にctxとして渡されます。"""
    return {
        "api_url": kwargs["api_url"],
        "timeout": kwargs.get("timeout_secs", 10),
        "client": _make_http_client(),
    }

def process(task: Any, **ctx: Any) -> dict[str, Any]:
    """1タスク毎に呼ばれます。task.pk + task.extraを元に処理します。
    戻りdictはruns.output_jsonに保存されます。例外を上げると失敗扱いになります。"""
    response = ctx["client"].get(f"{ctx['api_url']}/{task.pk}", timeout=ctx["timeout"])
    return {"status_code": response.status_code, "size": len(response.content)}

def cleanup(**ctx: Any) -> None:
    """プロセス停止時に呼ばれます (任意)。接続のクローズなどを行います。"""
    ctx["client"].close()

属性型意味
pkstrタスクの主キーです。
extradictenqueue時に添えたJSONです。
attemptint何回目の試行かを表します (0始まり)。
enqueued_atstr (ISO)キューに入った時刻です。

8. Executor種別

8.1 shell

任意のシェルコマンドを1タスク = 1プロセスとして実行します。

{
  "command": ["bash", "-c", "echo task={task.pk}"],
  "cwd": null,
  "env": {"MY_VAR": "value"},
  "timeout_secs": 60
}

テンプレート変数 {task.pk} {task.extra.KEY} が展開されます。

8.2 python_module

プラグインを呼びます。instanceはworkerプロセスでキャッシュされるため、重いsetup() は1回だけ実行されます。

{
  "source_path": "plugins/my_plugin",
  "module": "main",
  "init_kwargs": {
    "api_url": "https://api.example.com",
    "timeout_secs": 10
  }
}

10. 設定リファレンス

10.1制御プレーンCLI

pipeline run [OPTIONS]

  --host TEXT        listenするホスト名/IP (デフォルト0.0.0.0)
  --port INTEGER     listenポート (デフォルト8001)
  --db-url TEXT      SQLAlchemy URL (例: sqlite:///./pipeline.db)
  --log-level TEXT   debug / info / warning / error
  --mode TEXT        dev / prod (= CORS設定が変わります)

10.2ワーカーCLI

pipeline worker [OPTIONS]

  --control-url TEXT 制御プレーンのURL (例: http://10.0.0.1:8001) [必須]
  --worker-id TEXT   ワーカー識別子 (デフォルトhostname)
  --tags TEXT        カンマ区切り (例: gpu,cuda12)
  --token TEXT       制御プレーン認証トークン (任意)
  --log-level TEXT   ログレベル

10.3環境変数

変数意味
PIPELINE_DB_URL--db-url と同等です。
ADMIN_PASSWORD初回起動時に管理者パスワードを設定します (= 認証を有効化する場合)。

11. 運用Tips

11.1バックアップ

SQLiteファイル (pipeline.db) をそのままコピーすればフルバックアップになります。制御プレーンが動作中でもWALモードなので、読み出しロックは発生しにくいです。確実を期すなら VACUUM INTO をお使いください。

sqlite3 pipeline.db "VACUUM INTO 'pipeline-backup-$(date +%Y%m%d).db'"

11.2ワーカー追加 / 削除

ワーカーはpull型なので、追加はデーモン起動だけでOKです (= 制御プレーン側の設定変更は不要です)。削除はデーモン停止だけで、60秒後に lost となり、600秒後にDBから消えます。

11.3ワーカーのアフィニティ

workloadの host_affinity を設定すると、タグが一致するワーカーだけがclaimできます。例として ["gpu"] を指定するとtags=gpu 持ちのみが対象になります。

11.4失敗のリトライ

例外で失敗したタスクは max_attempts 回まで自動でretryされます。上限超過後はキューに残りますがclaimされなくなります (= dead-letter)。これらはDashboardのキュー深さで failed として見えます。手動で再投入するか、失敗原因を修正してenable切替で再開してください。

11.5監視

Prometheus形式のメトリクスendpointは未実装です (= 必要に応じて追加可能)。当面は以下の方法をおすすめします。

  • Dashboardのキュー深さで滞留を監視します。
  • /api/v1/dashboard/overview の recent_failures を定期pingします。
  • nvidia-smiメトリクスはワーカーから自動収集されます。
