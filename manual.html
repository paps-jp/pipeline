1. 用語と概念

用語説明
制御プレーン (Control plane) FastAPI + SQLite で動く 単一プロセス。 Web UI 配信 + REST API + ワーカー heartbeat 受付。
ワーカー (Worker) 各ホストで動く デーモン。 制御プレーンに heartbeat を 10 秒ごとに送り、 キューから タスクを 1 つずつ 取り出して処理。
ワークロード (Workload) 「何を どう実行するか」 の 定義。 slug で 一意。 Web UI から登録する 1 つの単位。
キュー (Queue) ワークロード毎に 1 つ作られる、 タスクの待ち行列。 SQLite 内の <slug>_queue テーブル。
タスク (Task) キューに 1 行入る 処理単位。 pk (主キー) + extra (任意の JSON) を持つ。
Run タスクの実行履歴 1 件。 成功 / 失敗 / 経過時間 / stdout / stderr / output_json が保存される。
Executor タスクを 実際に処理する方式。 shell (= 任意コマンド) と python_module (= プラグイン呼出) の 2 種類。
プラグイン Python モジュール 1 つを plugins/<name>/ に配置。 plugin.yaml + setup()/process()/cleanup() を export する。

1.1 タスクのライフサイクル

  1. enqueue — Web UI または POST API で キューに pk を 1 件追加 (= state pending)
  2. claim — 空きの ワーカーが claim_batch で 1 件取得 (= state claimed、 リース付き)
  3. process — ワーカーが プラグインの process(task) を呼ぶ
  4. complete / fail — 戻り値が dict なら成功 (= キュー行を削除 + runs に記録)、 例外なら失敗 (= attempt 増加 + state failed)
  5. retry — failed タスクは max_attempts 未満なら 再度 pending に戻る。 上限に達すると dead-letter として 残る。

2. Dashboard

制御プレーンの 全体状態を 一画面で 把握するためのページ。 3 秒ごとに更新。

パネル内容
実行中finished_at が null の run 一覧 (workload / pk / worker / 経過秒)
最近の失敗直近 300 件から success=false を 抽出 (最大 10 件)、 Python traceback の最終行を 表示
キュー深さenabled 各 workload の pending / claimed / failed をバーで可視化
システム情報version / mode (= dev/prod) / DB URL / 現在時刻
GPU 温度ワーカーが nvidia-smi メトリクスを heartbeat に乗せた場合、 過去 30 分の温度を スパークラインで 表示 (70℃ で 黄、 80℃ で 赤)

3. Workloads

ワークロードの 一覧 + 作成 + 編集。 各行に 「直近 20 件の 成否」 がスパークラインで 表示される (緑 = 成功 / 赤 = 失敗)。

3.1 ワークロード作成

「新規作成」 ボタンで Drawer が 右から開く。 入力項目:

項目説明
slugURL/識別子。 小文字英数 + ハイフン / アンダースコア。 一度作ったら 変更不可。
表示名UI 上での 人間可読名。
説明(任意) 何をする workload か メモ。
実行モードshell または python_module を選択。
実行設定shell ならコマンド配列、 python_module ならプラグイン + モジュール + 設定。
優先度0–100。 ワーカーが 複数 workload に対応する場合 高優先を 先に処理。
重み同優先度内での 配分比率 (= weighted fair queue)。
バッチサイズワーカーが 1 回の claim_batch で 取る件数。 大きいほど スループット 上昇、 小さいほど 公平性 / 失敗 局所化。
lease 秒数claim 後 この秒数 worker から complete/fail が来なければ 他 worker が 再 claim 可能。
max_attemptsfail → retry の上限回数。 超えると dead-letter として残る。

3.2 タスク投入

各行の 📨 アイコンで モーダルを開き、 pk と任意の extra (JSON) を入力して投入。 同じ pk は 重複扱いされ INSERT IGNORE される。

3.3 実行履歴

各行の 履歴アイコンで RunsDrawer が 右から開く。

  • キュー状態 (pending / claimed / failed) の バッジ表示
  • 直近 50 件の run を 表形式で 表示
  • 各行クリックで stdout / stderr / error / output_json を 展開 (= 失敗原因が一目で分かる)
  • 各行の 🔄 アイコンで 同じ pk を 再投入

3.4 URL deep link

各 workload に 直接 URL でアクセス可能:

  • /workloads/<slug> — その workload の 編集 Drawer が開く
  • /workloads/<slug>/runs — RunsDrawer が 開く

Slack や Issue に URL を貼る運用に 便利。

4. Workers

接続中ワーカー の一覧。 表示項目:

列意味
idワーカー識別子 (= 起動時に --worker-id で指定、 デフォルトは hostname)
hostワーカーの ホスト名
stateactive / draining / lost / connecting
current_workload現在処理中の workload (claim 中なら表示)
processed累計 処理タスク数
errors累計 エラー数
last_seen最後の heartbeat 経過時間
started_at起動時刻
state の遷移: heartbeat が 60 秒来ないと lost に。 さらに 600 秒来ないと DB から 削除される (= worker_metrics の 該当行も リーパーで 削除)。

5. ServiceLog

制御プレーン + 各ワーカーホスト の stdout/stderr を タブで切り替えて 表示。 動作中は teal の パルスバッジが 点滅、 一時停止中は灰色。

  • Pipeline タブ — 制御プレーン (= pipeline-oss-control.service) の ログ
  • 各 ホストタブ — Deploy で 登録した 各 配信先ホスト の ログ (= 配信時に該当ホストの service stdout/stderr が 制御プレーンに集約)

タブ切替時に 自動で 最下行 (= 最新) にスクロール。 「一時停止」 で 自動更新を 停止できる。

6. Deploy

制御プレーンから 各ホストへ ファイルやコードを 一括配信する 仕組み。 SSH を裏で使う が、 オペレータは Web UI だけで完結。

6.1 配信先ホスト

SSH 接続先の 一覧:

  • label: 表示名
  • host: IP または FQDN
  • ssh_user / ssh_port: 通常 root / 22
  • enabled: チェックを外すと この配信から 除外される

「公開鍵を表示」 で 制御プレーンが使う SSH 公開鍵を 表示。 これを 配信先ホストの ~/.ssh/authorized_keys に 入れることで パスワード無しの 配信が可能になる。

6.2 配信パス

「何を どこへ」 の 設定:

  • src_path: 制御プレーン上の パス (= 配信元)
  • dst_path: 配信先ホストでの パス (= 配信先)
  • setup_command: 配信直後に 1 回 実行するコマンド (任意)
  • service_command: service として常駐させる場合 のコマンド (任意)
  • delete_mode: rsync --delete を 付ける (= 余分なファイルを 配信先で 削除)

6.3 配信実行

「Deploy now」 ボタン → 有効化された 全配信先 × 全配信パスを 一括実行。

  • skip restart: rsync のみで service restart は 行わない
  • dry run: rsync --dry-run で 何が配信されるか だけを 表示

結果は 履歴に 残り、 各 run の ログを 展開して 確認できる。

7.3 main.py

from typing import Any

def setup(**kwargs: Any) -> dict[str, Any]:
    """プロセス起動時に 1 回 呼ばれる。 重い初期化 (モデルロード / DB 接続)
    はここで。 戻り dict が process() に ctx として 渡される。"""
    return {
        "api_url": kwargs["api_url"],
        "timeout": kwargs.get("timeout_secs", 10),
        "client": _make_http_client(),
    }

def process(task: Any, **ctx: Any) -> dict[str, Any]:
    """1 タスク毎に呼ばれる。 task.pk + task.extra を 元に処理。
    戻り dict は runs.output_json に保存される。 例外を上げると 失敗扱い。"""
    response = ctx["client"].get(f"{ctx['api_url']}/{task.pk}", timeout=ctx["timeout"])
    return {"status_code": response.status_code, "size": len(response.content)}

def cleanup(**ctx: Any) -> None:
    """プロセス停止時に呼ばれる (任意)。 接続のクローズなど。"""
    ctx["client"].close()

8. Executor 種別

8.1 shell

任意の シェルコマンドを 1 タスク = 1 プロセスとして 実行する。

{
  "command": ["bash", "-c", "echo task={task.pk}"],
  "cwd": null,
  "env": {"MY_VAR": "value"},
  "timeout_secs": 60
}

テンプレート変数 {task.pk} {task.extra.KEY} が 展開される。

8.2 python_module

プラグインを 呼ぶ。 instance は worker プロセスで キャッシュされる ため、 重い setup() は 1 回だけ実行。

{
  "source_path": "plugins/my_plugin",
  "module": "main",
  "init_kwargs": {
    "api_url": "https://api.example.com",
    "timeout_secs": 10
  }
}

10. 設定リファレンス

10.1 制御プレーン CLI

pipeline run [OPTIONS]

  --host TEXT        listen するホスト名/IP (デフォルト 0.0.0.0)
  --port INTEGER     listen ポート (デフォルト 8001)
  --db-url TEXT      SQLAlchemy URL (例: sqlite:///./pipeline.db)
  --log-level TEXT   debug / info / warning / error
  --mode TEXT        dev / prod (= CORS 設定が変わる)

10.2 ワーカー CLI

pipeline worker [OPTIONS]

  --control-url TEXT  制御プレーンの URL (例: http://10.0.0.1:8001) [必須]
  --worker-id TEXT    ワーカー識別子 (デフォルト hostname)
  --tags TEXT         カンマ区切り (例: gpu,cuda12)
  --token TEXT        制御プレーン認証トークン (任意)
  --log-level TEXT    ログレベル

10.3 環境変数

変数意味
PIPELINE_DB_URL--db-url と同等
ADMIN_PASSWORD初回起動時に 管理者パスワードを 設定 (= 認証を有効化する場合)

11. 運用 Tips

11.1 バックアップ

SQLite ファイル (pipeline.db) を そのままコピーすれば フル バックアップ。 制御プレーンが動作中でも WAL モードなので 読み出し ロックは 発生しにくい。 確実を期すなら VACUUM INTO:

sqlite3 pipeline.db "VACUUM INTO 'pipeline-backup-$(date +%Y%m%d).db'"

11.2 ワーカー追加 / 削除

ワーカーは pull 型なので、 追加は デーモン起動だけで OK (= 制御プレーン側の 設定変更 不要)。 削除は デーモン停止だけ、 60 秒後に lost、 600 秒後に DB から消える。

11.3 ワーカーの アフィニティ

workload の host_affinity を 設定する と、 タグが一致する ワーカーだけが claim できる。 例: ["gpu"] を指定すると tags=gpu 持ちのみ。

11.4 失敗の リトライ

例外で 失敗した タスクは max_attempts 回まで 自動 retry。 上限超過後は キューに残るが claim されなくなる (= dead-letter)。 これらは Dashboard の キュー深さで failed として見える。 手動で 再投入するか、 失敗原因を 修正して enable 切替で 再開。

11.5 監視

Prometheus 形式の メトリクス endpoint は 未実装 (= 必要に応じて 追加可能)。 当面は:

  • Dashboard の キュー深さで 滞留を 監視
  • /api/v1/dashboard/overview の recent_failures を 定期 ping
  • nvidia-smi メトリクスは ワーカーから 自動収集

11.6 アップデート

cd /opt/pipeline
git pull
pip install -e .
sudo systemctl restart pipeline-control
# 各 worker:
sudo systemctl restart pipeline-worker

Web UI を 更新した場合は cd web && npm install && npm run build も 実行。
